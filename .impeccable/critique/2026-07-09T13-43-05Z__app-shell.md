## Anti-Patterns Verdict

**Does this look AI-generated? No.** It reads as a real shadcn/Raycast-adjacent developer
utility: compact, neutral, task-first. The risk here is **product drift**, not decoration —
hidden hover actions, icon-only toolbars, inconsistent focus treatment, hard-coded status
colors, and a status bar that under-delivers on the "cockpit" promise.

**Deterministic scan (Assessment B):** Core shell files scanned **clean (exit 0)**. Only 2
findings across the wider tree, both false positives:
- `border-radius: 9999px` (main.css:190) — idiomatic pill radius for the scrollbar thumb. Not a violation.
- `fontFamily: Consolas` (terminal-view.tsx:42) — correct monospace for a terminal component.

Where the detector missed and the review caught: hard-coded `emerald-500` / `amber-500`
status colors (status-bar.tsx:18, detail-toolbar.tsx:177-193) bypass the token system — invisible
to the rule engine because they're Tailwind utility classes, but a real theme-consistency risk.

**Browser overlay:** unavailable — no browser-automation tool exposed and no dev server running.
No user-visible overlay was produced.

## Overall Impression

A trustworthy foundation, not yet a senior-grade shell. The bones are right — honest git
vocabulary, compact density, safe destructive flows, AI integrated as actions rather than a
chat takeover. The single biggest opportunity: **make the shell actively report truth**. The
bottom bar is mostly empty and primary actions hide behind discovery. A cockpit shows state
and controls at a glance; right now the developer has to hunt.

## What's Working

1. **Developer-native density** — 8/9-height controls, monospace branch/path labels, honest git language. Fits the product exactly.
2. **Destructive safety** — delete-worktree blocks on local changes / unreachable commits before confirming (delete-worktree-dialog.tsx:49-80).
3. **AI-forward without chat slop** — Copilot lives as actions/sessions, not a consumer-chat surface (detail-view.tsx:111-126).

## Priority Issues

### [P1] Status bar is not a cockpit
- **Why it matters:** The design promise ("show true state", North Star "Developer Cockpit") is persistent repo/branch/PR truth. Today the bar shows only background tasks or "Ready", leaving most of it empty (status-bar.tsx:52-66).
- **Fix:** Pass active workspace/worktree/repo/PR state into StatusBar; show active folder, branch/HEAD, dirty/ahead-behind, PR status, and task state as compact segments.
- **Suggested command:** /impeccable harden

### [P1] Primary actions are too hidden
- **Why it matters:** Power devs need one-glance actions. Terminal/Copilot/delete live behind context menus, dropdowns, and hover-only affordances (app-sidebar.tsx:276-358).
- **Fix:** Make the selected worktree's core actions explicitly visible; collapse only true overflow into one stable menu.
- **Suggested command:** /impeccable layout

### [P2] Keyboard/focus system is inconsistent
- **Why it matters:** Violates the design's keyboard-first 3px-ring rule. Sidebar uses ring-2; status bar hides the outline with no replacement (status-bar.tsx:47-49); focus can be invisible.
- **Fix:** Standardize focus-visible:ring-[3px] across custom buttons, sidebar actions, status bar, and segmented controls.
- **Suggested command:** /impeccable audit

### [P2] Component vocabulary drifts (settings)
- **Why it matters:** Settings uses rounded-2xl and shadow-sm/md hero cards (settings.tsx:51,145-150) — reads like a different product than the flat, tonal shell.
- **Fix:** Return settings to flat tonal panels, 8-10px radius, drop the icon-hero treatment.
- **Suggested command:** /impeccable polish

### [P2] Empty/loading states are too passive
- **Why it matters:** First-run home only says "Add a workspace from the sidebar" with no CTA (detail-view.tsx:83-88); loading is text-only (history.tsx:222-225).
- **Fix:** Add direct CTAs ("Add workspace", "Start Copilot session"); use skeletons where content shape is known.
- **Suggested command:** /impeccable onboard

## Persona Red Flags

**Alex (keyboard-first power user):** No visible shortcut map; core worktree actions require hover/menu discovery; no command palette to jump between workspaces/worktrees.

**Jordan (first-timer):** "Sessions" appears conditionally and may be non-clickable when empty (app-sidebar.tsx:393-394); empty home instructs "use the sidebar" instead of giving a button.

**Sam (accessibility-dependent):** Focus rings inconsistent and sometimes invisible; emerald/amber state colors are hard-coded, not tokenized — risky for contrast across the three themes and for color-blind users.

## Minor Observations

- Hard-coded emerald-500 / amber-500/600 status colors bypass tokens (status-bar.tsx:18, detail-toolbar.tsx:177-193).
- Branch/PR links are visually understated; useful state disappears into muted text.
- Toolbar title hierarchy is sometimes too quiet for the active workspace.

## Questions to Consider

- What should the bottom bar tell a developer in one second that the main pane does not?
- Why is Sessions not a first-class, stable nav destination if AI is first-class?
- Which three actions deserve permanent visibility for a selected worktree?
